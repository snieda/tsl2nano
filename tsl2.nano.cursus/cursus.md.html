Cursus: Repeatable Process
© Thomas Schneider 2018 ## Introduction Cursus is a change management system or change log using delta entries and timers in a scheduled process. Target is any store or database content that should be changed in a scheduled manner. The advantage is a history of changes and the possibility to undo/redo/replay the changes at any time. Cursus is totally independent to the target system (e.g. a database) and needs relation pathes plus object ids to reference the target entries. gets an amount of described change events (Consilium) that define a timer-period (Timer), the new value (Mutatio) and the target (Res). This change event descriptors will be sorted by timer.start and priority. Optional blockers (Obsidio) can avoid the excecution of change events. If a change event is activated and processed, optional impacts (Effectus) will be excecuted. The target result and the change event descriptor will be sealed and stored. Activated change events may be deactivated through the processor: * _Processor.deactivate(Set consiliums, Date from, Date until)_ * _Processor.resetTo(Set consiliums, Consilium lastActiveConsilium)_ ## JPA Entities For each element of Cursus it exists a jpa entity with a name starting with 'E'. Additionally an _EProcess_ (can start the Processor) and an _EProcessLog_ are defined. Each _EProcess_ needs a date-period for the change event descriptors to respect.
