Cursus: Repeatable Process
© Thomas Schneider 2018 ## Introduction Cursus is a change management system or change log using delta entries and timers in a scheduled process. Target is any store or database content that should be changed in a scheduled manner. The advantage is a history of changes and the possibility to undo/redo/replay the changes at any time. Cursus is totally independent of the target system (e.g. a database) and needs relation pathes plus object ids to reference the target entries. Cursus may be instrumented by insurances and other financial systems to build a course/track through changes on contracts. It is a kind of an implementation of the event source pattern. persisting the state of a business entity (like an Order or Customer) as a sequence of state-changing events, the application reconstructs an entity’s current state by replaying the events. ### Overview ![](http://g.gravizo.com/g?digraph G { Processor -> Consilium [label="Runs the process to activate\nscheduled change (Consilium) that define a timer-period (Timer), the new value (Mutatio) and the target (Res). This change event descriptors will be sorted by timer.start and priority. Optional blockers (Obsidio) can avoid the excecution of change events. If a change event is activated and processed, optional impacts (Effectus) will be excecuted. The target result and the change event descriptor will be sealed and stored. Activated change events may be deactivated through the processor: * _Processor.deactivate(Set consiliums, Date from, Date until)_ * _Processor.resetTo(Set consiliums, Consilium lastActiveConsilium)_ ## JPA Entities For each element of Cursus exists a jpa entity with a name starting with 'E'. Additionally an _EProcess_ (can start the Processor) and an _EProcessLog_ are defined. Each _EProcess_ needs a date-period for the change event descriptors to respect. To define a full change event descriptor (_EConsilium_) you need to crawl through: EConsilium -> EExsecutio -> EMutatio -> ERes ## Creating a path to the desired value to be changed A Mutatio (change) describes a path starting at the root object to the value to be changed The rules for the path are described in the class _ValuePath_: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ 1. customer.address[first].city 2. customer.address[0].city <- the first address 3. customer.address.city <- the first address 4. customer.address[last].street <- the last address 5. customer.address[-1].code <- the last address 6. customer.address[street=berlinerstrasse].city 7. customer.address[new].city <- create a new address 8. customer.address[?street=berlinerstrasse].city <- searches for given address, if not found, create a new one 9. customer.address[?2].city <- find the third address, or if not existing, create a new one 10 customer.address[?].city <- find the first address, or if not existing, create a new one ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ ## Usage / Examples To see a simple example for a financial organization with Contracts, Partners and Accounts see the junit test *CursusTests*. ## First Steps using it in NanoH5 * on the first login/connection dialog * select 'tsl2.nano.cursus-x.x.x-SNAPSHOT.jar' as _jar-file_ * select 'update' on _Auto-DDL_ * restart application * create a new change event: _EConsilium_ * create a new change execution: _EExecutio_ pointing to a new _EMutatio_ pointing to a new _ERes_ * the new "thing" (_ERes_) has to point to your desired Business Entity with a known Entity ID * create a new EProcess and start it (to run through the change) * create a new group (EGrex) using the existing _ERes_, let the validObjectIDs empty to run the process through all inactive consilii * press the button start -
